# Sahaya App Architecture & Developer Guide

This document outlines the architecture, data flows, and system components of the Sahaya Platform. Use this as a map for extending the Flutter applications and the Azure Flask backend.

---

## 🏗️ System Overview

Sahaya connects community issue-reporting with granular volunteer-driven task execution. It operates through three main system nodes:

1. **Sahaya Volunteer App (Flutter)**: For ground volunteers to find missions, update availability, route to tasks, and submit photo proofs.
2. **Sahaya NGO Admin App (Flutter)**: For NGO organizers to triage incoming field reports, verify volunteer proof, manage users, and orchestrate problem resolutions. 
3. **Sahaya FaaS API (Python Flask on Azure)**: Houses the heavy, asynchronous processing including the Telegram Webhook, AI-driven task decomposition, Algorithmic Volunteer Matching, and scheduled notification sweeps.

All state heavily relies on a central **Firebase (Firestore/Auth/Messaging)** hub, with heavy external delegation to **Google Gemini** for LLM cognitive tasks and **Cloudinary** for scalable media storage.

---

## 🗃️ Database Schema (Firestore)

The system relies on a NoSQL architecture containing several core collections:

*   `telegram_links`: Device-to-NGO mappings (`/register <NGO_ID>`).
*   `raw_uploads`: Dumping ground for unstructured data (images, voice, text) sourced from the Telegram webhook.
*   `problem_cards`: High-level community issues triaged from `raw_uploads`. Contains coordinates, descriptions, and a calculated `priorityScore`.
*   `tasks`: Bite-sized sub-tasks procedurally generated by AI from a parent `ProblemCard`. Includes required `skillTags`, `estimatedVolunteers`, etc.
*   `volunteer_profiles`: Global pool of registered volunteers holding location (`locationGeoPoint`), `radiusKm`, `skillTags`, and `availabilityWindowActive`.
*   `match_records`: Junction documents representing an invited or ongoing association between a Volunteer and a Task.
*   `ngo_notifications` / `volunteer_notifications`: In-app notification queues since push alerts can be unstable.

## 🧠 Core Engineering Flows

### 1. Ingestion Pipeline (Telegram -> Azure -> Firestore)
1. Field agents message the Sahaya Telegram bot. 
2. Azure `app.py` catches the webhook. Media binaries are extracted from Telegram's servers and uploaded statically to **Cloudinary** (converting Telegram limitations to stable URLs).
3. A `raw_uploads` document is written to Firestore mapped to the physical device's linked `ngoId`.
4. The NGO Admin's Flutter UI listens to `raw_uploads` and converts them into `ProblemCard`s using AI extraction.

### 2. Task Generation & Decomposition (Azure)
*   **Trigger**: An NGO Admin approves a raw `ProblemCard` via `review_detail_screen.dart`.
*   **API Segment**: Flutters POSTs to `/generate-tasks`.
*   **Action**: Azure prompts **Gemini 1.5 Flash** to absorb severity, location, and description, outputting 1 to 3 atomic volunteer tasks (e.g. *Technical Repair*, *Transport*). 
*   **Post-processing**: Priority algorithms evaluate severity + scale + recency to rank the `ProblemCard`, then Azure instantly daisy-chains into the Matching sweep.

### 3. Asymmetric Matching Algorithm (Azure)
*   **Trigger**: Follows `/generate-tasks` or manual calls targeting `/run-matching`.
*   **Action**: Azure pulls all locally active `volunteer_profiles`.
*   **Formula**: The Match Engine triangulates overlaps. 
    *   *Distance*: Haversine algorithm natively evaluates the GPS bounds between the Task and the Volunteer's operation radius.
    *   *Skills*: Overlap coefficient calculated.
    *   *Availability*: Rewards active presence.
*   **Result**: Top 20 best candidate scores generate new `match_records` holding `'status': 'open'`.

### 4. Mission Execution & Proof Loop (Flutter <-> Azure)
1. Volunteers observe matched tasks in their Active Missions feed. They use external maps to navigate to coordinates.
2. Upon completion, volunteers upload Proof Images (routed safely by Cloudinary pipeline).
3. The submission executes Azure `/notify-proof-submitted` to ping the NGO.
4. NGO Admin opens `proof_detail_screen.dart`.
    *   **AI Pre-check**: `gemini_service.dart` immediately fires the proof to Gemini alongside the task description. Gemini returns a verdict (e.g., *Likely Genuine* or *Needs Clarification*).
    *   **Caching**: Results are saved directly into the `match_record` to preserve bandwidth and bypass redundant API costs.
5. If Approved, `/complete-task` fires on Azure. It dynamically evaluates if `new_completion_count >= estimated_volunteers`, marking the task as `done` and automatically cascading upwards to close out the parent `ProblemCard` if all sibling tasks are fulfilled.

## 🤖 Feature Extensions: The Gemini Ecosystem

You utilize Gemini across several asynchronous architectures. When expanding this, respect these patterns:
*   **Backend (Python)**: Uses `google-generativeai`. Primarily handles background asynchronous synthesis (e.g., Task Decomposition JSON bridging, `whatToBring` generation, `impactStatement` summaries). Use strict regex extractors (`re.search(r'\[.*\]', ...)`) as basic `.replace()` methods are brittle against conversational padding.
*   **Frontend (Dart)**: Uses `google_generative_ai`. Empowers rich UI actions like the "Magic AI Editor" where admins can type "Double the volunteer count" and the AI naturally rips and restructures the JSON fields on the fly (`aiEdit` / `aiEditList`).

## 🚨 Guidelines for Future Development

### 1. Robust API Bridges
Your architecture leverages heavy asynchronous chaining (Firestore Stream -> Flutter UI -> HTTP Post -> Azure compute -> Firestore Update -> Flutter Stream updates). 
*   **Always use `.timeout(...)`** on Flutter HTTP posts. Azure FaaS is exceptionally vulnerable to cold-start delays. Unwrapped futures will crash your loaders and freeze users.
*   Never assume data shapes returning from Gemini perfectly match the schema. Pre-parse and validate strings via JSON boundary detection.

### 2. State & Security
*   **Firestore Rules** must enforce strict silo protocols. Ensure that `admin/NGO` documents restrict by `ngoId`, and volunteer profiles/matches restrict by `uid`.
*   When executing algorithmic loops like Availability Reminders (`/send-availability-reminders`), offload cron-heavy evaluations strictly to the native Python `APScheduler`. Do not rely on Flutter app lifecycles to trigger timed bulk dispatches!

### 3. Scaling the Matching Algorithm
Currently, matching performs memory evaluations across all volunteers. As the platform scales:
1. Shift to geospatial Firestore queries (e.g. `geoflutterfire` or Redis bounding boxes) before feeding documents into Python.
2. The current 20-match hard cap (`matches[:20]`) on the backend should evolve into a paginated radius sweep to avoid match spamming global volunteers.
